# Подключение Telegram-группы к отчётам бригадира

Сообщения из ТГ-группы автоматически становятся записями в карточке
объекта в системе. Бригадир продолжает писать как привык — а в системе
появляется структурированный отчёт. Никакого дублирования.

## Как это устроено

```
┌──────────────────┐
│ Бригадиры пишут  │
│ в форум-группу   │
│ (один топик =    │
│  один объект)    │
└────────┬─────────┘
         │ Telegram Bot API
         │ (long-polling от tg-bridge)
         ▼
┌────────────────────────────────────────┐
│ systemd: tg-bridge.service             │
│  · читает getUpdates (без webhook)     │
│  · парсит «Отчёт: / Объект: / *…»      │
│  · резолвит «Брусилова» → siteId       │
│  · POST в site-forms API               │
└────────────────────┬───────────────────┘
                     │ X-Deloresh-Write-Secret
                     ▼
┌────────────────────────────────────────┐
│ site-forms.service                      │
│ /var/lib/deloresh/site-forms/…/         │
│ brigadier-reports.json                  │
└────────────────────────────────────────┘
                     │
                     ▼
   В карточке объекта появляется
   «Отчёт бригадира» из ТГ.
```

## Поддерживаемый формат сообщений

Бот распознаёт сообщения такого вида (вариантивность шире, см. тесты в
`src/lib/tgReportParser.test.ts`):

```
Отчёт:
Объект: Брусилова
* Демонтаж тротуаров -250м2
* Выкоп траншеи под БК- 110мп
* Устройство щебня под БК- 50мп
* Прием материала
*Итр -2чел
*Количество рабочих- 16чел
* Техники- 7шт
```

Что превратится в системе в одну запись «Отчёт бригадира» с:

- **Объект:** Брусилова (`siteId=brusilova`)
- **Работы (lines):** 7 пунктов из звёздочек, без служебных строк
- **Ресурсы:** ИТР=2, рабочих=16, техники=7 — добавляются в поле комментария: «Ресурсы: ИТР: 2, Рабочих: 16, Техники: 7»
- **Ответственный:** имя автора сообщения в Telegram
- **Источник:** в комментарий добавляется маркер `[источник: Telegram]`
- **Время:** из метки сообщения

Сообщения, которые бот **игнорирует** (не пишет в систему):

- Без слова «Отчёт» в первых строках и без «Объект: …» — это обычная переписка.
- «Отчёт» есть, но имя объекта не сопоставляется с реестром — пишет
  предупреждение в журнал и не сохраняет (чтобы не плодить мусорные
  siteId).

## Подключение — пошагово

### Шаг 1. Создать бота в Telegram

1. Откройте чат с **@BotFather**.
2. Отправьте `/newbot`.
3. Придумайте имя (например, «Деловые Решения · Отчёты») и юзернейм
   (например, `deloresh_reports_bot`).
4. Получите токен вида `1234567890:AA…` — **сохраните его**.
5. Там же отправьте `/setprivacy` → выберите бота → **Disable**.
   Без этого шага бот видит только сообщения, в которых его упомянули
   через `@`.

### Шаг 2. Добавить бота в группу

1. В вашу форум-группу (или обычную группу) добавьте бота как
   участника.
2. **Сделайте его администратором** — иначе Telegram может урезать
   видимость сообщений в форум-топиках.
3. Никаких особых прав не нужно — только чтение сообщений.

### Шаг 3. Узнать chat_id (опционально)

Если хотите ограничить бота одной конкретной группой, получите её
`chat_id`. Запустите на сервере:

```bash
ssh deloresh '
  curl -s "https://api.telegram.org/bot<ВАШ_ТОКЕН>/getUpdates" | head -200
'
```

В ответе у объекта `chat` будет `id` вида `-1001234567890` (минус и
длинное число — это супергруппа). Запишите.

### Шаг 4. Положить env-файл на сервер

С вашего Mac:

```bash
ssh deloresh '
  test -f /etc/deloresh/tg-bridge.env || \
    install -m 600 /home/deploy/Delores-object/scripts/deploy/tg-bridge.env.example /etc/deloresh/tg-bridge.env
  ${EDITOR:-nano} /etc/deloresh/tg-bridge.env
'
```

Подставьте:

- `TG_BOT_TOKEN=…` (обязательно)
- `TG_ALLOWED_CHAT_ID=…` (опционально, можно указать несколько через запятую)

`SITE_FORMS_WRITE_SECRET` оставьте пустым — bootstrap сам подставит
значение из `/etc/deloresh/site-forms.env`.

### Шаг 5. Запустить bridge

```bash
npm run deploy:server -- root@94.242.58.24
```

Bootstrap увидит файл `/etc/deloresh/tg-bridge.env`, поставит unit
`tg-bridge.service`, подставит туда секрет и стартует сервис.

В выводе должно быть:

```
▸ Ставлю systemd unit tg-bridge (Telegram → site-forms)
● tg-bridge.service - Deloresh Objects — Telegram → site-forms bridge (бригадирские отчёты)
     Active: active (running)
```

### Шаг 6. Проверить, что работает

Напишите в группу тестовый отчёт. На сервере:

```bash
ssh deloresh 'journalctl -u tg-bridge -f'
```

В логе должно быть:

```
✓ бот: @deloresh_reports_bot (Деловые Решения)
+ отчёт tg-1001…-42 → siteId=brusilova (7 стр.)
```

Откройте `http://94.242.58.24/`, перейдите на карточку объекта
«Брусилова» → раздел «Отчёт бригадира» → видите ваше сообщение из
группы.

## Управление сервисом

```bash
# Логи в реальном времени
ssh deloresh 'journalctl -u tg-bridge -f'

# Перезапуск (например, после правки env)
ssh deloresh 'systemctl restart tg-bridge'

# Остановить (чтобы временно отключить мост)
ssh deloresh 'systemctl stop tg-bridge'

# Включить обратно
ssh deloresh 'systemctl start tg-bridge'

# Полностью убрать (но env-файл оставить)
ssh deloresh 'systemctl disable --now tg-bridge'
```

## Включение скачивания фото/видео

По умолчанию мост сохраняет только текст отчёта. Если хотите, чтобы
фото из ТГ автоматически попадали в фотофиксацию объекта:

```bash
ssh deloresh '
  sed -i "s|^TG_DOWNLOAD_MEDIA=.*|TG_DOWNLOAD_MEDIA=1|" /etc/deloresh/tg-bridge.env
  systemctl restart tg-bridge
'
```

Каждое фото из сообщения отчёта будет сохранено в `object-media`
объекта и подцеплено в SPA. Учтите, что это занимает место — см.
`docs/DEPLOY.ru.md`, раздел про объёмы хранения.

## Разбор частых случаев

### Бот в группе, но в логе тишина при отправке отчёта

- Проверьте Privacy Mode у бота: `@BotFather → /mybots → бот → Bot
  Settings → Group Privacy → Turn off`. После переключения **снова
  выгоните и добавьте бота в группу** — Telegram кэширует privacy
  state.
- Проверьте, что бот добавлен **админом** (особенно в форум-группе).
- Если задан `TG_ALLOWED_CHAT_ID` — убедитесь, что вы пишете в нужный
  чат (id отрицательный для супергрупп).

### Сообщение есть, но `! не удалось сопоставить «...» с объектом`

Имя в строке `Объект: …` не совпало ни с одним именем в реестре
объектов приложения. Варианты:

- В реестре нет такого объекта — добавьте через UI.
- Имя в ТГ написано по-другому — поправьте в сообщении или добавьте
  альтернативное имя в реестре (поле `name` объекта).
- Топик в форум-группе можно «прикрепить» к нужному объекту: бот при
  создании топика запоминает связь `thread_id ↔ siteId`, и
  последующие отчёты в этом топике резолвятся по топику, а не по
  тексту. Если связь не зафиксировалась автоматически — переименуйте
  топик так, чтобы имя совпало с объектом, и пересоздайте его.

### Отчёт сохранился дважды

Этого не должно случаться: бридж использует стабильный
`id = tg-<chatId>-<messageId>`, а API site-forms на дубликат отвечает
`{ok: true, duplicate: true}`. Если всё-таки видите две записи — это
значит, что я сохранил отчёт ДО запуска бриджа (вручную), а потом
бридж сохранил снова. Удалите дубль через интерфейс.
